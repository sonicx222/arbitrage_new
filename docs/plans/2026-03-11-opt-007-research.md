## 2. Proposed OPT-007 Phases

### Phase 1: Quick Wins (Low Risk, Measurable Impact)

#### 1A. PriceMatrix `writtenSlots` Set → Uint8Array Bitmap
**File**: `shared/core/src/caching/price-matrix.ts:247`

**Current**: `Set<number>` checked via `.has(index)` on every `getPrice()`, `getPriceOnly()`, `setPrice()`, `setBatch()`.
**Proposed**: Pre-allocated `Uint8Array` bitmap — `writtenSlots[index] !== 0` is a simple array access with no hash computation.

```typescript
// BEFORE: Set.has() involves hash computation + memory indirection
private writtenSlots: Set<number> = new Set();
// Check: this.writtenSlots.has(index)

// AFTER: Direct array access — O(1) with better cache locality
private writtenSlots: Uint8Array;  // allocated in constructor: new Uint8Array(totalSlots)
// Check: this.writtenSlots[index] !== 0
// Set:   this.writtenSlots[index] = 1
```

- **Impact**: Eliminates hash overhead on every price read. For 10K slots, bitmap = 10KB vs Set = ~200KB+.
- **Risk**: LOW — behavioral equivalent, simpler data structure.
- **Effort**: 0.5 hours. Touch: constructor, `setPrice()`, `setBatch()`, `getPrice()`, `getPriceOnly()`, `getStalePrice()`, `reset()`, plus tests.

#### 1B. GasPriceCache Stale Spread Elimination
**File**: `shared/core/src/caching/gas-price-cache.ts:471`

**Current**: `return { ...cached, isFallback: true }` creates a new object on every call when gas data is stale.
**Proposed**: Pre-compute a `staleVersion` when data transitions to stale, or set `isFallback` on the mutable cache entry itself.

```typescript
// BEFORE: Spread on every stale read (~1000s/sec during stale periods)
return { ...cached, isFallback: true };

// AFTER: Mutate in place when staleness detected (one-time)
if (!cached.isFallback) {
  cached.isFallback = true;
}
return cached;
```

- **Impact**: Eliminates object allocation during stale gas periods.
- **Risk**: LOW — `isFallback` is a read-only flag downstream. Must reset to `false` on next successful refresh.
- **Effort**: 0.5 hours.

#### 1C. SnapshotManager: Use Cached BigInt Reserves
**File**: `services/unified-detector/src/detection/snapshot-manager.ts:116-121`

**Current**: `BigInt(pair.reserve0)` re-parses the string on every snapshot creation — even though `pair.reserve0BigInt` is already set on the runtime object (via `chain-instance.ts:1981`).
**Proposed**: Add `reserve0BigInt?: bigint` to `ExtendedPair` interface and prefer it.

```typescript
// BEFORE: String→BigInt reconversion (wasteful: chain-instance already did BigInt→string)
reserve0BigInt = BigInt(pair.reserve0);

// AFTER: Use cached BigInt if available (zero allocation)
reserve0BigInt = pair.reserve0BigInt ?? BigInt(pair.reserve0);
```

- **Impact**: Eliminates BigInt parse per snapshot × 2 fields. During high-activity periods (perpetual cache invalidation), this runs 100-1000×/sec per chain.
- **Risk**: LOW — fallback to string parse for backward compat.
- **Effort**: 0.5 hours. Add fields to `ExtendedPair` in snapshot-manager.ts, update `createPairSnapshot()`.

#### 1D. Root tsconfig: ES2020 → ES2022
**File**: `tsconfig.json:5`

**Current**: `"target": "ES2020"` — all service tsconfigs already override to ES2022, but root uses ES2020.
**Proposed**: Align root with `"target": "ES2022"`, `"lib": ["ES2022"]`. Node 22 supports ES2024; ES2022 is conservative and safe.

- **Impact**: Better codegen: native class fields (no WeakMap polyfills), `Array.at()`, `Object.hasOwn()`, top-level await.
- **Risk**: VERY LOW — all services already build at ES2022. This aligns root.
- **Effort**: 0.25 hours. Change 2 lines in `tsconfig.json`.

### Phase 2: Detection Data Layout

#### 2A. Eliminate `BigInt.toString()` in `applyReserveUpdate`
**File**: `services/unified-detector/src/chain-instance.ts:1979-1980`

**Current**: Every Sync event does `pair.reserve0 = reserve0BigInt.toString()` — 2 string allocations per event. These strings feed into `PairSnapshot.reserve0` (string) which is consumed by `getDexPools()` and whale alerts.

**Proposed**: Defer `toString()` to the cold paths that actually need strings (snapshot creation, whale alerts), not the hot Sync handler.

```typescript
// BEFORE: Always convert (hot path)
pair.reserve0 = reserve0BigInt.toString();  // allocation every Sync event

// AFTER: Lazy conversion via getter or compute in snapshot
pair.reserve0BigInt = reserve0BigInt;
// pair.reserve0 is only updated when a snapshot is created (cold path)
```

- **Impact**: Saves 2 string allocations per Sync event × 100-1000/sec/chain × 15 chains = 3000-30000 allocations/sec eliminated.
- **Risk**: MEDIUM — requires verifying all consumers of `pair.reserve0` (string). Downstream: snapshot-manager, whale-alert publisher, logging. Must ensure lazy conversion is thread-safe.
- **Effort**: 2-3 hours. Touch: `applyReserveUpdate()`, `ExtendedPair` type, `createPairSnapshot()`, tests.
- **Dependency**: Phase 1C must land first (snapshot manager uses BigInt directly).

#### 2B. SnapshotManager Per-Pair Invalidation
**File**: `services/unified-detector/src/detection/snapshot-manager.ts:267-270`

**Current**: `invalidateCache()` nulls the entire snapshot cache on every Sync event. Since Sync events arrive at 100-1000/sec, the 100ms TTL cache is perpetually stale during active trading. The cache only helps during quiet periods.

**Proposed**: Track a per-pair `dataVersion` and check it during snapshot reads. Only regenerate snapshots for pairs whose data changed since last cache hit.

```typescript
// Option A: Version-gated per-pair freshness (simple)
invalidatePair(pairAddress: string): void {
  this.dirtyPairs.add(pairAddress);
  this.snapshotVersion++;
}

// In createPairsSnapshot: only rebuild dirty pairs, reuse cached entries for clean pairs
```

- **Impact**: Reduces snapshot object creation from O(all pairs) to O(dirty pairs) per check. With 500 pairs and 1-2 updating per Sync event, this is a ~250x reduction in snapshot allocations during high-activity periods.
- **Risk**: MEDIUM — more complex cache logic. Must ensure correctness when multiple pairs update in the same block.
- **Effort**: 3-4 hours. Touch: SnapshotManager, chain-instance `applyReserveUpdate`, tests.

### Phase 3: PriceMatrix Read Optimization

#### 3A. `getPrice()` Reusable Result Object
**File**: `shared/core/src/caching/price-matrix.ts:700-703`

**Current**: Returns `{ price, timestamp }` — new object per call.
**Proposed**: Callers that process synchronously can use a shared mutable result object.

```typescript
// Add alongside existing getPrice():
private readonly _priceResult: PriceEntry = { price: 0, timestamp: 0 };

getPriceInto(key: string, out: PriceEntry): boolean {
  // ... same logic, writes into 'out' instead of allocating
  out.price = price;
  out.timestamp = absoluteTimestamp;
  return true;
}
```

- **Impact**: Zero-allocation price reads for callers that opt in. The existing `getPrice()` API is unchanged.
- **Risk**: LOW — additive API, callers opt in.
- **Effort**: 1 hour. Add method, update detection loops to use it.
- **Alternative**: Promote `getPriceOnly()` usage where timestamp isn't needed (already exists at line 760).
