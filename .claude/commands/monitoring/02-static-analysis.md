## Incremental Mode

```bash
LAST_SHA=$(cat ./monitor-session/last-run.sha 2>/dev/null || echo "")
CURRENT_SHA=$(cat ./monitor-session/current.sha)

if [ -n "$LAST_SHA" ] && [ "$LAST_SHA" != "unknown" ] && [ "$LAST_SHA" != "$CURRENT_SHA" ]; then
  git diff --name-only "$LAST_SHA".."$CURRENT_SHA" > ./monitor-session/changed-files.txt
  CHANGED_COUNT=$(wc -l < ./monitor-session/changed-files.txt)
  echo "INCREMENTAL MODE: $CHANGED_COUNT files changed since last run ($LAST_SHA)"
  INCREMENTAL=true
else
  echo "FULL SCAN MODE: No previous run SHA or first run"
  INCREMENTAL=false
fi
```

When `INCREMENTAL=true`, checks marked `[INC]` can be narrowed to only changed files.
Checks marked `[FULL]` always scan the full codebase (global invariants).
If no relevant files changed for an `[INC]` check, skip with an INFO finding.

### 1C -- MAXLEN Enforcement [INC]

Every XADD call must include MAXLEN trimming to prevent unbounded stream growth.

**Part 1 -- Config level:**
1. Read `STREAM_MAX_LENGTHS` from `shared/core/src/redis/streams.ts`.
2. Verify all streams in `inventory.json` have a MAXLEN value.

**Part 2 -- Call-site level:**
3. Grep `this\.xadd\(` and `this\.xaddWithLimit\(` in `.ts` under `shared/` and `services/` (exclude `node_modules`, tests).
4. For raw `this.xadd()` not inside `xaddWithLimit()` itself, check if MAXLEN option is passed.
5. Also search for `redis.xadd(` or `client.xadd(` bypassing `RedisStreamsClient`.

Flags:
- Stream missing from `STREAM_MAX_LENGTHS` -> H:MAXLEN
- Raw `this.xadd()` not using `xaddWithLimit()` -> H:MAXLEN
  - Watch: `shared/core/src/redis/streams.ts` ~line 876 (HMAC rejection DLQ path)
- Direct `redis.xadd()` bypassing abstraction -> C:MAXLEN

Note: `StreamBatcher` and `xaddWithLimit()` apply MAXLEN via `STREAM_MAX_LENGTHS`.
Only `xaddWithLimit()` is safe; raw `xadd()` bypasses enforcement.

### 1F -- Nullish Coalescing Anti-Pattern [INC]

1. Grep `\|\| 0\b` and `\|\| 0n\b` in `.ts` files (exclude `node_modules`).
